**审核报告：xsgrok2 v3.1 正文内容质量问题**

## 一、问题根因分析

从资深小说编辑（专注网文质量把控10年以上）的视角来看，三大问题本质上都源于**“上下文记忆的碎片化与状态缺失”**这一核心矛盾。

1. **情节或文字重复率高**  
   根本原因在于**“无长期记忆 + 机械式全量注入”**。当前架构每次都把完整大纲、全部Lorebook、世界设定塞进prompt，而前文记忆仅保留前一章1500字。当小说进入中后期（第8章以后），模型无法有效记住前文关键事件，只能依赖prompt中反复出现的“大纲关键词”，导致AI进入“复读机模式”。此外，temperature 0.85偏高，在缺乏“情节新颖性约束”的情况下，模型倾向于输出高概率的套路描写（例如反复出现“心神巨震”“瞳孔地震”等），形成段落级重复。缺少章节摘要机制，使得每一章生成都像是“独立创作”，自然会与前面章节情节撞车。

2. **人物逻辑或对话逻辑错误**  
   根源是**缺乏结构化的角色状态机（Character State Machine）**。目前角色信息以纯文本形式全量注入，属于“一次性描述”，模型无法持续追踪角色在故事进程中的“已知信息”“情绪存续”“关系演变”“目标更新”。当prompt长度爆炸后，系统提示里的“角色行为必须符合设定”这条规则被严重稀释。ChapterInstruction中的`characterChanges`虽然存在，但仅作为一次性指令，未形成持久化状态，导致第N章AI“忘记”了第N-2章角色经历的创伤或承诺，出现性格崩坏。

3. **称呼错误**  
   这是人物逻辑错误的具象化表现。称呼系统本质上是**动态的关系映射**（甲对乙的称呼会随亲密度、场合、情绪变化），而当前系统完全依赖模型的“瞬时记忆”。当上下文不足或prompt中同时出现多个角色时，模型极易发生**关系错位**（把当前对话对象当成记忆中的另一个角色）。Lorebook全量注入反而加剧了这个问题——过多无关人名干扰了注意力分配。

总结：所有问题都指向同一个架构缺陷——**当前系统是“一次性提示词堆砌机”，而非“具备长期记忆和状态追踪的小说生成状态机”**。

## 二、现有架构缺陷

从内容质量控制专业角度，当前架构存在以下致命缺陷：

- **无分层记忆体系**：仅有“短期记忆（前一章1500字）+ 全局记忆（全量设定）”，缺失中长期记忆（章节摘要、人物状态快照、情节事件链）。
- **数据模型过于扁平**：`Novel`和`Chapter`均为纯文本字段，缺少`ChapterSummary`、`CharacterStateSnapshot`、`RelationshipMatrix`、`PlotPointTracker`等结构化字段，无法支持一致性校验。
- **Prompt工程过于粗放**：GrokRepository.kt中采用“把所有东西塞进去”的策略，未采用RAG（Retrieval-Augmented Generation）、动态Prompt组装、Self-Consistency校验等现代技术。
- **缺少生成后校验关卡**：生成即入库，没有“AI自我批判+编辑修正”流程，无法拦截低质输出。
- **重复检测与节奏控制缺失**：无任何基于向量或规则的重复检测机制，也无叙事节奏模板（张力曲线）约束。

## 三、改进方案

### P0（必须立即实施）

**方案1：引入结构化记忆层（Memory Hierarchy）**  
**目标**：解决AI“失忆”和状态漂移问题。  
**具体实现步骤**：
1. 在数据库新增三张表：`ChapterSummary`（摘要+关键事件点）、`CharacterState`（每个角色在每章结束时的状态JSON：情绪、目标、已知信息、关系值）、`PlotTracker`（已触发情节点集合）。
2. 修改GrokRepository.kt的prompt构建逻辑：**不再全量注入大纲和所有Lorebook**，改为注入“当前最相关的前3章摘要 + 当前所有角色最新状态快照 + 本章指令”。
3. 生成新章节后，立即调用另一个Prompt让模型生成“本章摘要”和“角色状态更新”，持久化保存。
**预期效果**：人物逻辑一致性提升60%以上，跨章情节重复率大幅下降。

**方案2：双Pass生成 + Self-Critique机制**  
**目标**：拦截重复和逻辑错误。  
**具体实现步骤**：
1. 将生成流程改为两阶段：First Pass（生成正文）+ Second Pass（批判修正）。
2. Second Pass的系统提示词必须极其严格，包含：“请以资深责编身份检查以下问题：1. 情节是否与前文摘要重复？2. 角色行为是否符合最新状态？3. 所有称呼是否正确？”并要求模型输出<revision>标签内的修改后文本。
3. 在Android端增加“生成质量评分”字段，低于85分的章节不允许直接进入正文列表。
**预期效果**：可直接过滤70%的低质内容，显著降低称呼错误和逻辑bug。

### P1（下一迭代重点）

**方案3：动态称呼映射表 + 关系矩阵**  
**目标**：根治称呼错误。  
**具体实现步骤**：
1. 在`CharacterState`中增加`addressMap`字段（JSON），格式为`{ "targetCharacterId": { "currentCall": "师父", "formality": 0.8, "affection": -0.3 } }`。
2. 生成对话时，强制在Prompt中注入当前说话者和听者的最新`addressMap`。
3. 每次角色状态更新时，要求模型同步更新称呼映射。
**预期效果**：称呼错误率降低至接近0。

**方案4：RAG-based Lorebook + 情节向量去重**  
**目标**：解决prompt过长和重复问题。  
**具体实现步骤**：
1. 将Lorebook向量化（使用本地embedding模型或Grok API embedding），生成时仅检索与本章指令最相关的3-5条。
2. 建立已生成内容向量库，在生成前对“本章创作指令+前文摘要”做相似性检索，若与已存在章节相似度>0.75，则强制降低temperature并加入“必须创新”指令。
**预期效果**：减少无关信息干扰，显著降低文字和情节重复率。

### P2（持续优化）

**方案5：叙事节奏模板与张力曲线控制**  
**目标**：提升整体节奏感，减少注水和重复描写。  
**具体实现步骤**：在ChapterInstruction中增加`rhythmCurve`字段（信息量/张力曲线），生成Prompt中要求模型严格按曲线分配高潮与铺垫段落。

## 四、与其他角色的协作建议

1. **Prompt Engineer（提示词工程师）**：需共同重构系统提示词，将当前“规则堆砌”改为“角色扮演+Few-shot+批判性思考”三段式结构。我提供编辑视角的“审稿 checklist”，对方转化为Prompt。
2. **Backend Developer**：必须优先支持新增的`CharacterState`和`ChapterSummary`表结构，以及生成后的自动状态更新流程。这是所有P0方案的基础。
3. **Product Manager**：需要重新定义“章节合格”的标准，不能仅看字数，应增加“一致性得分”“重复率得分”“称呼准确率”等量化指标，并体现在UI上。
4. **测试/QA**：建议建立“一致性测试集”（专门构造容易让AI出错的复杂关系网和长线伏笔），用于回归测试新记忆系统效果。
5. **数据标注人员**：初期可人工标注10本高质量小说的“章节摘要模板”和“角色状态快照模板”，用于Few-shot学习和模型微调方向指导。

以上方案若能按P0→P1顺序落地，xsgrok2的内容质量将实现从“可用”到“可商用”的本质跨越。
